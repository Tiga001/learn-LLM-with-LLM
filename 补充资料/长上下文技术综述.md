## 二、RoPE 数学基础（理解所有后续方法的前提）

### 2.1 RoPE 的核心公式

RoPE（Rotary Position Embedding）将位置信息编码为向量的旋转角度。

对于位置 $m$ 处的向量 $x \in \mathbb{R}^d$，RoPE 将其每两个维度 $(x_{2i}, x_{2i+1})$ 视为一个二维向量，施加一个角度为 $m\theta_i$ 的旋转：

$$
\text{RoPE}(x_m, m) = \begin{pmatrix} x_{2i} \cos(m\theta_i) - x_{2i+1} \sin(m\theta_i) \\ x_{2i} \sin(m\theta_i) + x_{2i+1} \cos(m\theta_i) \end{pmatrix}
$$

其中频率由维度索引 $i$ 决定：

$$
\theta_i = \text{base}^{-2i/d}, \quad i = 0, 1, \ldots, d/2 - 1
$$

默认 $\text{base} = 10000$。

用复数形式更优雅地表达：

$$
f(x_m, m) = x_m \cdot e^{im\theta}
$$

### 2.2 为什么 RoPE 有效

RoPE 的关键性质是**相对位置编码**——两个位置 $m$ 和 $n$ 之间的注意力分数只取决于相对距离 $m - n$：

$$
\langle f(q, m), f(k, n) \rangle = \text{Re}\left[q \cdot \bar{k} \cdot e^{i(m-n)\theta}\right]
$$

这意味着模型学到的是"两个 token 相距多远"，而不是"token 在哪个绝对位置"。

### 2.3 频率的直觉：高频 vs 低频

RoPE 的 $d/2$ 个维度对应不同的频率 $\theta_i$：

```
维度 i=0 (最低维度):
  θ_0 = 10000^{0} = 1
  → 旋转角度 = m × 1    → 旋转很快 → 高频
  → 位置每变化 1，角度变化 1 弧度
  → 擅长区分近距离 token (如 m=5 和 m=6)

维度 i=d/2-1 (最高维度):
  θ_{d/2-1} = 10000^{-1} = 0.0001
  → 旋转角度 = m × 0.0001 → 旋转极慢 → 低频
  → 位置变化 10000 才旋转 1 弧度
  → 擅长区分远距离 token (如 m=0 和 m=10000)

类比:
  高频维度 ≈ 时钟的秒针 → 精确计量短时间
  低频维度 ≈ 时钟的时针 → 粗略计量长时间
  
  要区分相隔 3 秒的两个时刻 → 看秒针
  要区分上午和下午       → 看时针
```

### 2.4 频率分布的可视化

以 $d = 128$（即 64 个频率维度）为例：

```
频率索引 i:    0        16       32       48       63
θ_i:          1.0      0.1      0.01     0.001    0.0001
波长 λ_i:     2π       63       628      6283     62832
              ↑                                    ↑
           高频区                                低频区
         区分近距离                            区分远距离
         
波长 λ_i = 2π / θ_i = 完成一个完整旋转周期所需的位置数
  高频: λ ≈ 6 个 token → 6 个 token 内旋转一圈
  低频: λ ≈ 63000 个 token → 63000 个 token 才转一圈
```

**这个频率分布是理解所有 RoPE 扩展方法的基础。**

## 三、外推问题：为什么超过训练长度就崩了

### 3.1 外推失败的数学解释

假设模型训练时最长序列为 $L_{\text{train}} = 4096$。

在训练中，每个频率维度 $i$ 见过的旋转角度范围是：

$$
m \cdot \theta_i \in [0, \; L_{\text{train}} \cdot \theta_i)
$$

对于**高频维度**（$\theta_i \approx 1$）：

$$
\text{角度范围} = [0, 4096) \text{ 弧度} \approx 652 \text{ 圈}
$$

高频维度已经转了很多圈，函数 $\sin$ 和 $\cos$ 是周期的，所以即使 $m > 4096$，高频维度的值仍然在已见过的范围内。**高频维度天然支持外推。**

对于**低频维度**（$\theta_i \approx 0.0001$）：

$$
\text{角度范围} = [0, 0.41) \text{ 弧度} \approx 0.065 \text{ 圈}
$$

低频维度在整个训练过程中只转了不到 $1/15$ 圈！

当推理时 $m = 16384$（4 倍外推）：

$$
m \cdot \theta_i = 16384 \times 0.0001 = 1.64 \text{ 弧度} \approx 0.26 \text{ 圈}
$$

这个角度值远超训练时见过的 $[0, 0.41)$ 范围。模型在这个角度上从未被训练过，注意力分数会产生异常值。

### 3.2 直觉理解

```
训练时低频维度:
  旋转角度: [0°, 23°]  ← 模型只见过这个范围
  
推理时 4× 外推:
  旋转角度: [0°, 94°]  ← 超出训练范围!
                          23° 到 94° 之间是"未知领域"
  
类比:
  你只学过加法 1+1 到 100+100
  现在让你算 400+400
  
  对于"100 以内"的加法 (高频): 你已经很熟练了
  对于"超过 100"的加法 (低频): 你可能外推对了 (因为加法是线性的)
  
  但 sin/cos 不是线性的!
  → 低频维度在未见过的角度范围上行为不可预测
```

### 3.3 外推失败的实验现象

```
LLaMA-2 (训练长度 4096) 在不同推理长度下的 PPL:

序列长度:     2K    4K    5K    6K    8K    16K
PPL:          5.2   5.4   18    150   >1000  爆炸

关键观察:
  → 超过训练长度后 PPL 急剧上升
  → 不是"慢慢变差"，而是"断崖式崩溃"
  → 因为 sin/cos 在未知角度上产生了完全随机的注意力分数
```

## 四、Position Interpolation（PI）— 最朴素的解法

### 4.1 核心思想

既然问题是"推理位置超出了训练范围"，那最简单的方法就是：**把推理位置压缩回训练范围。**

$$
m' = m \times \frac{L_{\text{train}}}{L_{\text{target}}}
$$

### 4.2 数学推导

原始 RoPE 频率：$\theta_i = \text{base}^{-2i/d}$

PI 等价于将频率缩小：

$$
\theta_i^{\text{PI}} = \theta_i \times \frac{L_{\text{train}}}{L_{\text{target}}} = \frac{L_{\text{train}}}{L_{\text{target}}} \cdot \text{base}^{-2i/d}
$$

也可以理解为修改了 base：

$$
\text{base}_{\text{PI}} = \text{base} \times \left(\frac{L_{\text{target}}}{L_{\text{train}}}\right)^{d/2}
$$

### 4.3 具体例子

```
目标: 将 LLaMA-2 从 4K 扩展到 16K (4× 扩展)

缩放因子 s = L_train / L_target = 4096 / 16384 = 0.25

推理位置 m=8192:
  PI 后: m' = 8192 × 0.25 = 2048
  → 2048 在 [0, 4096) 范围内 → 安全!

推理位置 m=16384:
  PI 后: m' = 16384 × 0.25 = 4096
  → 4096 正好是训练范围的边界 → 安全!

每个位置之间的角度差:
  原始:  Δ(m·θ) = 1 × θ_i     ← 相邻 token 的角度差
  PI 后: Δ(m·θ) = 0.25 × θ_i  ← 角度差缩小了 4 倍!
```

### 4.4 PI 的问题

```
问题: 所有频率被均匀缩放 → 高频维度被过度压缩

                原始              PI (4× 扩展)
高频 (i=0):     Δθ = 1.0          Δθ = 0.25      ← 区分度下降 4 倍!
中频 (i=32):    Δθ = 0.01         Δθ = 0.0025     ← 区分度下降 4 倍
低频 (i=63):    Δθ = 0.0001       Δθ = 0.000025   ← 区分度下降 4 倍

高频维度本来就能外推 (已经转了很多圈)，不需要被压缩!
但 PI 还是把它压缩了 → 近距离 token 的区分度白白降低了

这就好像:
  时钟的秒针本来好好的，你非要把它转慢 4 倍
  → 现在 4 秒钟才走 1 格 → 精度下降了
  → 但时针本来就该被调慢 → 只该调时针，别动秒针!
```

### 4.5 PI 的效果

- 需要少量微调（约 1000 steps）才能恢复质量
- 4× 扩展效果好，8× 以上开始明显下降
- 论文：*Extending Context Window of Large Language Models via Positional Interpolation*（Meta, 2023）

## 五、NTK-aware RoPE — 只缩放该缩放的频率

### 5.1 核心洞察

PI 的问题是"一刀切"地缩放所有频率。NTK-aware 的关键洞察来自信号处理的 Nyquist 采样定理：

```
高频维度 (θ ≈ 1):
  波长 λ = 2π/θ ≈ 6 个 token
  在 4096 个 token 中已经完成了 ~652 个周期
  → 完全不需要缩放, 它自带"外推能力"

低频维度 (θ ≈ 0.0001):
  波长 λ = 2π/θ ≈ 63000 个 token
  在 4096 个 token 中只完成了 0.065 个周期
  → 必须缩放, 否则推理时会超出已见过的角度

NTK-aware 的方法:
  不缩放频率本身, 而是修改 base
  → 让低频自然变得更低 (更大的波长)
  → 高频几乎不受影响
```

### 5.2 数学推导

NTK-aware 修改 base：

$$
\text{base}_{\text{NTK}} = \text{base} \times \alpha
$$

其中缩放因子：

$$
\alpha = \left(\frac{L_{\text{target}}}{L_{\text{train}}}\right)^{d/(d-2)}
$$

代入频率公式：

$$
\theta_i^{\text{NTK}} = (\text{base} \times \alpha)^{-2i/d}
= \text{base}^{-2i/d} \times \alpha^{-2i/d}
= \theta_i \times \alpha^{-2i/d}
$$

**关键**在于 $\alpha^{-2i/d}$ 这个因子对不同维度 $i$ 的影响不同：

```
当 i=0（最高频维度）:
  缩放因子 = α^{0} = 1         ← 完全不缩放!

当 i=d/2-1（最低频维度）:
  缩放因子 = α^{-(d-2)/d} ≈ α^{-1} ≈ L_train / L_target  ← 缩放最大

中间维度:
  缩放因子平滑过渡
```

### 5.3 对比 PI vs NTK-aware

```
                     PI                    NTK-aware
高频 (i=0):          缩放 0.25 (损失!)      缩放 1.0 (无损!)
中低频 (i=32):       缩放 0.25             缩放 ~0.5 (适中)
低频 (i=63):         缩放 0.25             缩放 ~0.25 (和 PI 类似)

视觉化 (4× 扩展):
  
  频率缩放因子
  1.0 |  ●                          ← NTK: 高频保持
      |   ●
      |    ●  
  0.5 |     ●               
      |      ●            
      |       ●           
  0.25| ●●●●●●●●●●●●●●     ●       ← PI: 所有频率统一缩放
      |                      ●
  0.0 |________________________●___
      i=0    i=16   i=32   i=48  i=63
      高频 ←──────────────────→ 低频
```

### 5.4 为什么不需要微调

NTK-aware 高频维度几乎不受影响。模型在推理时：

- 近距离 token 的区分度保持不变（高频正常工作）
- 远距离 token 能编码更大的位置（低频被拉长）
- 已有的注意力模式大部分保留

所以可以做到 **zero-shot**（不微调直接用），这是 NTK-aware 相对 PI 最大的实践优势。

### 5.5 代码实现

```python
import torch
import math

def ntk_aware_rope_freqs(dim: int, base: float = 10000.0,
                         max_position: int = 4096,
                         target_position: int = 16384) -> torch.Tensor:
    """计算 NTK-aware RoPE 的频率"""
    scale = target_position / max_position
    alpha = scale ** (dim / (dim - 2))
    new_base = base * alpha
    freqs = 1.0 / (new_base ** (torch.arange(0, dim, 2).float() / dim))
    return freqs

# 对比
dim = 128
freqs_orig = 1.0 / (10000 ** (torch.arange(0, dim, 2).float() / dim))
freqs_ntk = ntk_aware_rope_freqs(dim, target_position=16384)
freqs_pi = freqs_orig * (4096 / 16384)

print("维度  |  原始频率   |   PI频率   | NTK频率  | PI缩放 | NTK缩放")
print("-" * 70)
for i in [0, 8, 16, 32, 48, 63]:
    print(f"  {i:3d}  | {freqs_orig[i]:.6f} | {freqs_pi[i]:.6f} | "
          f"{freqs_ntk[i]:.6f} | {freqs_pi[i]/freqs_orig[i]:.4f} | "
          f"{freqs_ntk[i]/freqs_orig[i]:.4f}")
```

## 六、Dynamic NTK — 按需缩放

### 6.1 问题

固定的 NTK-aware 有一个缺陷：

```
假设 target_position = 16384

当推理序列实际只有 2000 tokens 时:
  → 远没有超出训练长度 4096
  → 但 base 已经被永久修改了
  → 低频维度被不必要地拉伸了
  → 可能对短序列产生微小的质量损失
```

### 6.2 核心思想

Dynamic NTK 的解决方案很简单：**根据当前实际序列长度动态计算 base**。

```python
def dynamic_ntk_base(current_seq_len: int, max_train_len: int = 4096,
                     dim: int = 128, base: float = 10000.0) -> float:
    """根据当前序列长度动态计算 NTK base"""
    if current_seq_len <= max_train_len:
        return base
    
    scale = current_seq_len / max_train_len
    alpha = scale ** (dim / (dim - 2))
    return base * alpha
```

### 6.3 行为分析

```
序列长度 ≤ 4096:
  → base 不修改 → 和原始 RoPE 完全一致 → 短序列零损失

序列长度 = 8192 (2× 外推):
  → α 较小 → base 轻微放大 → 温和的频率调整

序列长度 = 16384 (4× 外推):
  → α 较大 → base 显著放大 → 强力频率调整

序列长度 = 32768 (8× 外推):
  → α 更大 → base 更大 → 更强的频率调整

特点: 缩放程度"恰好够用", 不多不少
```

### 6.4 Dynamic NTK 的优势

| 对比项 | 固定 NTK | Dynamic NTK |
|--------|---------|-------------|
| 短序列 (< L_train) | 有微小损失 | **零损失** |
| 适度外推 (2-4×) | 效果好 | **效果好** |
| 大幅外推 (8×+) | 可能不够 | **自动加大力度** |
| 实现复杂度 | 极低 | 极低（多一个 if） |

## 七、YaRN — 当前最优的 RoPE 扩展方法

### 7.1 YaRN 的三个核心创新

YaRN（*Yet another RoPE extensioN*，Peng et al., 2023）综合了 PI 和 NTK 的优点，并加入了自己的改进。

#### 创新 1：NTK-by-parts（分区缩放）

YaRN 把 RoPE 的频率维度分成三个区域，对每个区域施加不同的缩放策略：

$$
\theta_i^{\text{YaRN}} = \begin{cases}
\theta_i & \text{if } \lambda_i < \lambda_{\text{low}} \quad \text{(高频区: 不缩放)} \\
(1-\gamma)\cdot\frac{\theta_i}{s} + \gamma\cdot\theta_i & \text{if } \lambda_{\text{low}} \le \lambda_i \le \lambda_{\text{high}} \quad \text{(过渡区: 插值)} \\
\theta_i / s & \text{if } \lambda_i > \lambda_{\text{high}} \quad \text{(低频区: 完全缩放)}
\end{cases}
$$

其中 $\lambda_i = 2\pi / \theta_i$ 是波长，$s = L_{\text{target}} / L_{\text{train}}$ 是扩展比例，$\gamma$ 是过渡区的插值权重。

```
YaRN 的分区策略:

    缩放因子
    1.0  |  ●●●●●●●                          ← 高频: 完全不动
         |         ●
         |          ●                         ← 过渡区: 平滑插值
    0.5  |           ●
         |            ●
         |             ●
    1/s  |              ●●●●●●●●●●            ← 低频: PI 式完全缩放
         |_________________________________
         i=0          过渡区          i=63
         高频 ←────────────────────→ 低频

对比 NTK-aware:
    NTK:  所有维度连续缩放 (无明确分区)
    YaRN: 明确的三区划分 + 过渡
    → YaRN 对高频维度的保护更彻底
```

分区边界由两个超参数控制：

- $\lambda_{\text{low}}$：高频区边界（默认 = 1，即波长小于 1 的维度不缩放）
- $\lambda_{\text{high}}$：低频区边界（默认 = 训练长度的某个比例）

#### 创新 2：注意力温度修正

当位置编码被缩放后，注意力分数的方差也会变化。YaRN 引入温度修正因子 $\sqrt{t}$：

$$
\text{Attention}(Q, K) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k} \cdot \sqrt{t}}\right)V
$$

其中 $t$ 根据扩展比例自动计算：

$$
t = \frac{1}{1 + \frac{1}{2s} \ln s}
$$

**直觉**：位置编码被拉伸后，不同位置之间的角度差变小了，导致注意力分数变得更平坦（softmax 变"软"）。温度修正通过调整 softmax 的"锐度"来补偿这一点。

#### 创新 3：极少量微调

```
YaRN 微调量:
  PI:    需要 ~1000 steps
  YaRN:  只需要 ~400 steps (约 0.1% 训练数据)
  
  4K → 128K 扩展 (32×), 只需 400 步微调
  → 模型质量几乎无损
```

### 7.2 YaRN 代码实现

```python
import torch
import math

def yarn_rope_freqs(
    dim: int,
    base: float = 10000.0,
    max_train_len: int = 4096,
    target_len: int = 65536,
    beta_fast: int = 32,
    beta_slow: int = 1,
) -> torch.Tensor:
    """YaRN 的 NTK-by-parts 频率计算"""
    scale = target_len / max_train_len
    
    freqs = 1.0 / (base ** (torch.arange(0, dim, 2).float() / dim))
    wavelengths = 2 * math.pi / freqs
    
    low_bound = max_train_len / beta_fast
    high_bound = max_train_len / beta_slow
    
    yarn_freqs = torch.empty_like(freqs)
    for i in range(len(freqs)):
        wl = wavelengths[i].item()
        if wl < low_bound:
            yarn_freqs[i] = freqs[i]
        elif wl > high_bound:
            yarn_freqs[i] = freqs[i] / scale
        else:
            gamma = (wl - low_bound) / (high_bound - low_bound)
            yarn_freqs[i] = (1 - gamma) * (freqs[i] / scale) + gamma * freqs[i]
    
    return yarn_freqs
```

### 7.3 YaRN 效果

| 扩展倍数 | PI (PPL) | NTK-aware (PPL) | YaRN (PPL) |
|---------|---------|-----------------|------------|
| 2× (4K→8K) | 5.8 | 5.9 | **5.5** |
| 4× (4K→16K) | 7.2 | 8.1 | **6.0** |
| 8× (4K→32K) | 12.5 | 15.3 | **6.8** |
| 16× (4K→64K) | >100 | >50 | **8.2** |
| 32× (4K→128K) | 爆炸 | 爆炸 | **10.5** |

YaRN 在大幅扩展时的优势尤为明显。

## 八、LongRoPE — 搜索最优缩放因子

### 8.1 核心思想

前面的方法（PI、NTK、YaRN）都是用**人工设计的公式**来决定每个频率维度的缩放因子。

LongRoPE（Microsoft, 2024）的思路完全不同：**用搜索算法找到每个维度的最优缩放因子。**

$$
\theta_i^{\text{LongRoPE}} = \lambda_i \cdot \theta_i
$$

其中 $\lambda_i$（$i = 0, 1, \ldots, d/2-1$）是每个维度独立的缩放因子，通过进化搜索（Evolutionary Search）在验证集上优化得到。

### 8.2 搜索空间

```
参数空间:
  每个频率维度 i 都有一个独立的缩放因子 λ_i
  搜索范围: λ_i ∈ [1/s, 1]  (其中 s = 扩展倍数)
  
  维度数: d/2 = 64 (对于 d=128)
  → 64 维连续搜索空间

搜索目标:
  min_{λ_0, ..., λ_{63}} PPL(model with rescaled RoPE)

搜索方法:
  进化搜索 (Evolutionary Search)
  → 随机初始化一组候选方案
  → 评估每个方案的 PPL
  → 保留最优 + 变异产生新方案
  → 迭代直到收敛
  
  搜索成本: ~数小时 (无需训练, 只需前向评估)
```

### 8.3 LongRoPE 的发现

通过搜索，LongRoPE 发现的最优缩放因子**并不符合任何简单的数学公式**：

```
搜索得到的缩放因子 (示意):
  
    λ_i
    1.0  |  ●●  ●●●                         
         |    ●     ●  ●                    
         |               ●                  
    0.5  |                 ●  ●             
         |                     ●●           
         |                       ●          
    1/s  |                        ●  ●●●●●  
         |_________________________________
         i=0                          i=63
         
  → 不是简单的单调递减
  → 某些维度的最优缩放因子出乎意料
  → 这就是人工设计公式的局限: 真正的最优解更复杂
```

### 8.4 LongRoPE 的两阶段策略

```
阶段 1: 搜索 + 短微调
  目标: 从 4K 扩展到 256K
  方法: 搜索最优缩放因子 → 微调 ~1000 步
  
阶段 2: 再次搜索（在微调后的模型上）
  目标: 从 256K 进一步扩展到 2048K (200万 tokens!)
  方法: 在阶段 1 模型上再搜索一组新的缩放因子
  
效果:
  LLaMA-2-7B: 4K → 2048K (512× 扩展), PPL 仍然合理
  在长文档任务上大幅超越 YaRN
```

## 九、工程方案：Sliding Window Attention

### 9.1 核心思想

上面的方法都在解决"位置编码外推"的问题。Sliding Window Attention 换了一个思路：**直接限制每个 token 只关注最近 $W$ 个 token。**

$$
\text{Attention}(q_t) = \text{softmax}\left(\frac{q_t K_{[\max(0,t-W):t]}^T}{\sqrt{d_k}}\right) V_{[\max(0,t-W):t]}
$$

### 9.2 有效感受野

```
单层的感受野 = W (窗口大小)

但 Transformer 是多层堆叠的!

Layer 0: token t 直接看到 [t-W, t]
Layer 1: token t 的表示已包含 Layer 0 中 [t-W, t] 的信息
          Layer 1 再看 [t-W, t]
          → token t 间接包含了 [t-2W, t] 的信息

Layer L: 有效感受野 = L × W

Mistral-7B: L=32, W=4096
  有效感受野 = 32 × 4096 = 131072 ≈ 128K
  但 KV Cache 只需存 W=4096 tokens!

这就像传话游戏:
  每个人只能听到身边 4 个人的话
  但经过 32 轮传话, 最远的人也能间接收到消息
  
  代价: 信息经过多次"转述", 远距离信息会衰减和失真
```

### 9.3 KV Cache 优势

```
标准 Full Attention 的 KV Cache:
  KV Cache 大小 = O(seq_len)  → 随序列长度线性增长
  seq_len=128K → KV Cache 极其庞大

Sliding Window 的 KV Cache:
  KV Cache 大小 = O(W)  → 固定, 与序列长度无关!
  W=4096 → 无论序列多长, KV Cache 始终只有 4096 个位置

  实现方式: 环形缓冲区 (Ring Buffer)
  ┌──────────────────────┐
  │ pos 0 │ pos 1 │ ... │ pos W-1 │
  └──────────────────────┘
  当新 token 到来时, 覆盖最旧的位置:
    写入位置 = current_pos % W
```

### 9.4 优缺点

| 维度 | Full Attention | Sliding Window |
|------|---------------|----------------|
| 单层感受野 | 整个序列 | W 个 token |
| KV Cache | $O(N)$, 随序列增长 | $O(W)$, 固定 |
| 长距离信息 | 直接捕获 | 间接经多层传递, 有衰减 |
| 是否需要特殊训练 | 否 | 是（训练时就要用 SWA） |
| 代表模型 | LLaMA, GPT | Mistral-7B |

## 十、StreamingLLM — 无限长度的流式推理

### 10.1 Attention Sink 现象

StreamingLLM（Xiao et al., 2024）发现了一个出人意料的现象：

```
在自回归 LLM 中, 前几个 token (尤其是第 1 个 BOS token) 
会获得极高的注意力权重, 即使它们在语义上毫无意义

例如一句话 "[BOS] The cat sat on the mat"
各位置获得的注意力权重:
  [BOS]:  0.35   ← 语义无关, 但权重极高!
  "The":  0.03
  "cat":  0.05
  "sat":  0.15
  "on":   0.02
  "the":  0.05
  "mat":  0.35   ← 当前 token, 权重高是正常的
```

**为什么会这样？** 根本原因是 softmax 的数学性质：

```
softmax 要求: 所有权重之和 = 1

假设一个位置不需要关注任何历史 token (比如该位置本身就包含足够信息):
  → 模型想让所有注意力权重趋近 0
  → 但 softmax 做不到! 总和必须是 1
  → 所以模型把"多余"的注意力集中分配给某个固定位置
  → 前几个 token (尤其是 BOS) 成了"垃圾桶", 吸收多余注意力
  → 这就是 Attention Sink

验证: 如果把前几个 token 的 KV Cache 删掉
  → softmax 的权重分布被打乱
  → 模型输出质量断崖式下降
```

### 10.2 Window Attention 为什么会崩

```
纯 Window Attention (只保留最近 W 个 token):

序列:  [BOS] t1 t2 ... t_{W} t_{W+1} ...
       ↑ attention sink

当序列超过 W 后, BOS 被踢出窗口:
  → attention sink 消失
  → softmax 分布失去"垃圾桶"
  → 注意力分数变得异常
  → PPL 瞬间飙升 (从 ~5 飙到 >10000)

PPL 变化:
  序列长度:    W/2     W       W+1      2W
  PPL:         5.2     5.4     >1000    >10000
                                ↑ 断崖! sink 被踢出的瞬间
```

### 10.3 StreamingLLM 的策略

```
核心: 保留 Attention Sinks + 最近窗口

保留前 k 个 token 的 KV (attention sinks, 通常 k=4)
+ 保留最近 w 个 token 的 KV (滑动窗口)

完整序列:
  [t1, t2, t3, t4, t5, t6, ..., t_{N-w}, ..., t_N]
   ↑ sinks (k=4) ↑    ↑ 中间全部丢弃 ↑   ↑ 最近窗口 ↑

StreamingLLM 实际 KV Cache:
  [t1, t2, t3, t4, t_{N-w+1}, ..., t_N]
   ↑    sinks   ↑  ↑     recent window    ↑

总 KV Cache 大小 = k + w = 常量!
  → 与序列长度完全无关
  → 可以处理无限长的流式输入
```

### 10.4 StreamingLLM 效果

```
LLaMA-2 (训练长度 4096):

方法                  序列 4K    序列 20K    序列 4M     是否稳定
─────────────────────────────────────────────────────────────
Full Cache            5.4       OOM         OOM         ✗
Window (W=4096)       5.4       >10000      >10000      ✗ (超窗口就崩)
StreamingLLM (k=4)    5.4       5.6         5.6         ✓ (始终稳定!)
```

### 10.5 StreamingLLM 的本质局限

```
StreamingLLM 不是真正的长上下文理解!

它做到的:
  ✓ 固定显存, 无限长度推理
  ✓ PPL 保持稳定
  ✓ 不需要任何微调

它做不到的:
  ✗ 回忆窗口外的任何信息
  ✗ 长文档问答 (问一个在 10000 token 前出现的事实)
  ✗ 需要全局信息的任务 (全文摘要)
  
本质区别:
  长上下文技术 (YaRN等) = 让模型真正"看到"更远的信息
  StreamingLLM = 让模型在"有限记忆"下不崩溃
  
  StreamingLLM ≈ 金鱼记忆 + 稳定运行
  长上下文 ≈ 大象记忆 + 全局理解
  
适用场景:
  ✓ 持续对话 (chatbot)
  ✓ 实时字幕/翻译
  ✓ 流式日志分析
  ✗ 长文档 QA
  ✗ 代码仓库理解
```

## 十一、LongLoRA — 低成本训练长上下文

### 11.1 问题

直接用长序列微调一个 4K 模型到 64K 是非常昂贵的：

```
Full Attention 的训练成本:
  序列长度 4K  → 注意力计算量 ∝ 4K² = 16M
  序列长度 64K → 注意力计算量 ∝ 64K² = 4096M
  
  计算量暴增 256 倍!
  
  而且长序列需要更大的 GPU 显存来存储激活值
  → 单卡可能根本放不下
```

### 11.2 LongLoRA 的两大创新

#### 创新 1：Shifted Sparse Attention (S²-Attn)

```
核心: 训练时用廉价的稀疏注意力, 推理时切回完整注意力

标准 Full Attention (seq_len = 8192):
  每个 token 与所有 8192 个 token 做 attention
  → 计算量 O(N²) = O(8192²)

S²-Attn (group_size G = 2048):
  Step 1 — 分组注意力 (一半注意力头):
    序列分成 4 组, 每组内部做 attention
    [t0..t2047] [t2048..t4095] [t4096..t6143] [t6144..t8191]
       组内 attn    组内 attn      组内 attn      组内 attn
    
    每组计算量 = O(G²), 共 N/G 组
    总计算量 = (N/G) × O(G²) = O(N × G) = O(N²/4)  ← 缩小 4 倍!
  
  Step 2 — 偏移分组注意力 (另一半注意力头):
    序列偏移 G/2 = 1024 后再分组:
    [t1024..t3071] [t3072..t5119] [t5120..t7167] [t7168..t1023(wrap)]
       组内 attn      组内 attn      组内 attn       组内 attn
    
    偏移使得 Step 1 中的组边界处的 token 在 Step 2 中被连接起来
    → 信息可以跨组流动!

  两步合并: 每个 token 都能间接看到整个序列的信息
  总计算量 ≈ O(N²/G × 2) << O(N²)

关键: S²-Attn 只在训练时使用!
     推理时切回 Full Attention + FlashAttention
     → 推理质量完全不受影响
     → 训练成本大幅降低
```

#### 创新 2：改进的 LoRA

```
标准 LoRA: 只在 QKV 投影和 MLP 上加 LoRA adapter
  → 对长上下文微调效果不够好

LongLoRA 的改进: 额外在 embedding 层和 normalization 层也加上可训练参数
  → Embedding: 位置编码相关, 对长上下文至关重要
  → Normalization: 影响数值稳定性
  → 总可训练参数只增加很少, 但效果提升明显
```

### 11.3 效果

```
LLaMA-2-7B 扩展到 100K 上下文:
  Full Fine-tuning:  需要 128 × A100 GPU, 训练几天
  LongLoRA:          只需 8 × A100 GPU, 训练几小时
  
  质量: LongLoRA ≈ Full Fine-tuning (差距 <1 PPL)
  成本: LongLoRA ≈ Full Fine-tuning 的 1/16
```

## 十二、Ring Attention — 跨多 GPU 的超长序列

### 12.1 问题

当序列真的长到百万级 token 时，**单卡根本放不下 KV Cache 和激活值**，必须分布式计算。

```
LLaMA-7B, 序列长度 1M tokens, FP16:
  KV Cache ≈ 2 × 32 × 1M × 128 × 2 bytes ≈ 32 GB (单层!)
  32 层总共 ≈ 32 × 32 = 1024 GB ← 远超任何单卡!
```

### 12.2 Ring Attention 的核心思想

```
将序列分片到多个 GPU 上, 用环形通信流水线实现 attention

假设 4 个 GPU, 序列长度 N = 8192, 每个 GPU 分 2048 tokens:
  GPU 0: tokens [0, 2048)     → 本地 Q_0, K_0, V_0
  GPU 1: tokens [2048, 4096)  → 本地 Q_1, K_1, V_1
  GPU 2: tokens [4096, 6144)  → 本地 Q_2, K_2, V_2
  GPU 3: tokens [6144, 8192)  → 本地 Q_3, K_3, V_3

问题: GPU 0 的 Q_0 需要与所有 GPU 的 K, V 做 attention
     但 K_1, K_2, K_3 在其他 GPU 上!

解决方案: 环形传递 KV 块 + 本地 FlashAttention

  Round 0:
    GPU 0: Attn(Q_0, K_0, V_0)  ← 本地计算
    GPU 1: Attn(Q_1, K_1, V_1)
    GPU 2: Attn(Q_2, K_2, V_2)
    GPU 3: Attn(Q_3, K_3, V_3)
    同时: K_0→GPU1, K_1→GPU2, K_2→GPU3, K_3→GPU0 (环形传递)

  Round 1:
    GPU 0: Attn(Q_0, K_3, V_3)  ← 收到 GPU 3 的 KV
    GPU 1: Attn(Q_1, K_0, V_0)  ← 收到 GPU 0 的 KV
    GPU 2: Attn(Q_2, K_1, V_1)
    GPU 3: Attn(Q_3, K_2, V_2)
    同时: 继续环形传递 KV

  Round 2: ...
  Round 3: ...
  
  共 4 轮后, 每个 GPU 都与所有 KV 完成了 attention!

关键:
  - 每轮的本地计算和 KV 传递是重叠的 (通信隐藏在计算中)
  - 使用 Online Softmax 在各轮之间正确合并部分结果
  - 与 FlashAttention 完美兼容
```

### 12.3 Ring Attention 的优势

```
序列长度扩展:
  单 GPU:    受限于单卡显存 → 最长 ~128K
  Ring Attn: 序列长度 ∝ GPU 数量
             8 GPU → 8× 序列长度
             64 GPU → 64× 序列长度 → 百万级 token!

通信开销:
  每轮传递 KV 块大小 = O(N/P × d), P 为 GPU 数
  传递轮数 = P-1
  总通信量 = O(N × d)
  
  当 N 足够大时, 计算量 O(N²d) >> 通信量 O(Nd)
  → 通信几乎完全被计算隐藏
  → 近线性的扩展效率!
```

## 十三、其他稀疏注意力方案（简述）

### 13.1 Longformer

```
Longformer (Beltagy et al., 2020):
  结合两种注意力模式:
  
  1. Local Attention: 每个 token 只关注窗口内的 w 个 token
  2. Global Attention: 特定 token (如 [CLS]) 可以关注所有位置
  
  复杂度: O(N × w) + O(N × g) ≈ O(N), 其中 g 是 global token 数
  
  适用: 长文档分类、问答
  局限: 需要预先指定 global token, 不适合自回归生成
```

### 13.2 BigBird

```
BigBird (Google, 2020):
  结合三种注意力模式:
  
  1. Local Attention: 窗口注意力
  2. Global Attention: 全局 token
  3. Random Attention: 随机采样远距离 token
  
  理论: 随机+局部+全局 = 图的通用近似器
  复杂度: O(N)
  
  适用: 长文档理解
  局限: 随机模式难以高效实现, 自回归场景不常用
```

### 13.3 多尺度/层级注意力

```
一些模型在不同层使用不同的窗口大小:
  底层: 小窗口 (精确的局部特征)
  高层: 大窗口 (粗粒度的全局特征)
  
  例如某些架构:
    Layer 0-7:   Window = 1024
    Layer 8-15:  Window = 4096
    Layer 16-23: Window = 16384
    Layer 24-31: Full Attention

  → 在显存和效果之间取得更好的平衡
```

## 十四、全面对比与选型指南

### 14.1 所有方案汇总对比

| 方案 | 类别 | 核心思想 | 是否需要微调 | 推理额外开销 | 最大扩展倍数 | 质量影响 |
|------|------|---------|------------|------------|------------|---------|
| **Position Interpolation** | 位置编码 | 均匀压缩位置 | 需要 (~1000步) | 无 | 4-8× | 小（高频被压缩） |
| **NTK-aware RoPE** | 位置编码 | 修改 base, 保留高频 | 不需要 | 无 | 4-8× | 小-中 |
| **Dynamic NTK** | 位置编码 | 动态修改 base | 不需要 | 极低 | 4-8× | 小 |
| **YaRN** | 位置编码 | 分区缩放+温度修正 | 极少 (~400步) | 极低 | 4-32× | 最小 |
| **LongRoPE** | 位置编码 | 搜索最优缩放因子 | 少量 (~1000步) | 无 | 64-512× | 最小 |
| **LongLoRA** | 训练方法 | S²-Attn + LoRA | 需要 (几小时) | 无（推理用Full） | 8-32× | 小 |
| **Sliding Window** | 注意力架构 | 固定窗口 | 训练时即用 | 无 | 理论无限 | 窗口外衰减 |
| **StreamingLLM** | 推理策略 | Sink + Window | 不需要 | 无 | 无限（流式） | 窗口外完全丢失 |
| **Ring Attention** | 分布式 | 环形传递 KV | 不需要 | 通信开销 | ∝ GPU 数 | 无（精确等价） |

### 14.2 场景选型指南

```
场景 1: "已有 4K 模型, 想零成本扩展到 16K"
  → Dynamic NTK (最简单, 无需微调, 短序列无损)
  → NTK-aware (如果不需要对短序列的严格兼容)

场景 2: "已有 4K 模型, 愿意少量微调, 扩展到 64K+"
  → YaRN (最佳质量-成本平衡, 只需 ~400 步微调)
  → LongRoPE (愿意花几小时搜索, 可达 256K+)

场景 3: "要训练一个新的长上下文模型 (64K-128K)"
  → 直接用长数据训练 + Sliding Window (如 Mistral 的做法)
  → LongLoRA (预算有限时, 用 S²-Attn 降低训练成本)

场景 4: "要部署一个持续对话的 chatbot (不限长度)"
  → StreamingLLM (固定显存, 无限长度)
  → 但注意: 用户不能让 bot 回忆很久以前的对话内容

场景 5: "要做长文档 QA / 全书摘要 (需要全局理解)"
  → YaRN / LongRoPE + Full Attention
  → StreamingLLM 不适合!
  → 如果文档超长: Ring Attention (多 GPU)

场景 6: "序列超过百万 tokens"
  → Ring Attention (唯一的选择, 需要多 GPU)
  → 配合 LongRoPE / YaRN 解决位置编码问题
```

## 十五、面试高频问题

```
Q1: "RoPE 为什么会外推失败?"
  → 低频维度在训练中只旋转了不到 1 圈
  → 推理时超出训练长度, 低频角度进入未见过的范围
  → sin/cos 在新角度上产生不可预测的注意力分数
  → 高频维度天然不受影响 (已经转了很多圈)

Q2: "PI 和 NTK-aware 的核心区别是什么?"
  → PI: 均匀缩放所有频率 → 高频被不必要地压缩 → 近距离区分度下降
  → NTK: 通过修改 base, 高频几乎不变、低频被缩放 → 近距离精度保持
  → PI 需要微调, NTK 可以 zero-shot

Q3: "YaRN 比 NTK-aware 好在哪?"
  → NTK: 所有维度连续缩放, 高频也有轻微影响
  → YaRN: 三区划分 (不缩放/过渡/全缩放), 高频保护更彻底
  → YaRN 还加了注意力温度修正, 补偿 softmax 分布变化
  → 大倍数扩展 (16×+) 时 YaRN 优势显著

Q4: "StreamingLLM 和长上下文技术有什么本质区别?"
  → StreamingLLM = 无限长 + 有限记忆 (金鱼模式)
  → 长上下文 = 有限长 + 完整记忆 (大象模式)
  → StreamingLLM 不能回忆窗口外的信息, 不适合全局任务
  → 它解决的是"不崩溃"而不是"理解更远"

Q5: "Sliding Window Attention 的有效感受野怎么算?"
  → 单层: W (窗口大小)
  → L 层后: L × W
  → Mistral: 32 × 4096 = 128K
  → 但远距离信息经多层传递会衰减, 不如 Full Attention 精确

Q6: "LongLoRA 的 S²-Attn 为什么只在训练时用?"
  → S²-Attn 是近似 (分组注意力), 会损失精度
  → 训练时用它: 降低计算成本, 让模型学会处理长序列
  → 推理时切回 Full Attention: 获得最高质量
  → 因为 FlashAttention 在推理时已经很高效了

Q7: "Ring Attention 怎么做到通信隐藏?"
  → 计算量 O(N²d/P²) per round, 通信量 O(Nd/P) per round
  → 当 N 足够大: 计算 >> 通信
  → 用异步通信 (NCCL) 与本地 FlashAttention 重叠
  → 通信时 GPU 同时在算上一轮收到的 KV 块
```
