# Day 4：DeepSeek-R1 论文精读 — 纯 RL 训练的推理涌现

> **目标**：精读 DeepSeek-R1 技术报告——理解 R1-Zero 如何在不使用 SFT 数据的情况下通过纯 RL 训练（GRPO）让模型自发涌现出 Chain-of-Thought、反思与自我纠正能力；掌握 R1 的完整四阶段训练管线（cold start SFT → RL → rejection sampling + SFT → RL）；深入理解 GRPO 在 R1 中的具体应用与 reward 设计；分析 R1 的数据蒸馏策略如何将大模型的推理能力转移到小模型。本日是 Day 5 GRPO 算法推导的实践动机来源。

## 一、R1 的定位与背景

### 1.1 论文核心信息

- **标题**：*DeepSeek-R1: Incentivizing Reasoning Capability in LLMs via Reinforcement Learning*
- **作者**：DeepSeek-AI (2025)
- **核心贡献**：证明纯 RL 训练可以让 LLM 自发涌现推理能力，并提出完整的训练管线使 R1 在数学、编程、推理任务上达到 o1 级别

### 1.2 核心问题

R1 试图回答一个根本性问题：

> **推理能力（如 Chain-of-Thought）是否必须通过 SFT 数据教给模型？还是可以通过 RL 自发涌现？**

传统观点：需要大量的 CoT 标注数据来教模型「如何思考」。
R1 的发现：**纯 RL 训练就能让模型自发学会推理**——这是论文最震撼的发现。

### 1.3 与 OpenAI o1 的关系

| 维度 | OpenAI o1 | DeepSeek-R1 |
|------|----------|-------------|
| 发布时间 | 2024 年 9 月 | 2025 年 1 月 |
| 是否开源 | 否 | 是 |
| 训练方法细节 | 未公开 | 完整公开 |
| 核心技术 | 推测：大规模 RL + CoT 数据 | GRPO + 多阶段训练 |
| 推理能力 | AIME 2024: 83.3% | AIME 2024: 79.8%（接近） |
| 模型规模 | 未知 | 671B (MoE) |

R1 的重要性在于：它是第一个**完整公开训练细节**的 o1 级别推理模型。

## 二、DeepSeek-R1-Zero：纯 RL 的惊人发现

### 2.1 实验设计

R1-Zero 的实验设计极其简洁：

```
起点：DeepSeek-V3-Base（预训练模型，未经 SFT）
训练方法：GRPO（Group Relative Policy Optimization）
训练数据：数学/编程/逻辑推理 prompt（无 SFT 数据！）
Reward：规则验证的准确性 + 格式奖励
```

关键：**没有任何 SFT 阶段**，没有任何 CoT 标注数据。直接在预训练模型上跑 RL。

### 2.2 Reward 设计

R1-Zero 使用了极其简单的 reward：

| Reward 类型 | 规则 | 值 |
|------------|------|-----|
| **准确性奖励** (Accuracy Reward) | 验证最终答案是否正确 | +1（正确）/ -1（错误） |
| **格式奖励** (Format Reward) | 检查输出是否包含 `<think>` 和 `</think>` 标签 | +1（符合格式）/ 0（不符合） |

$$r(x, y) = r_{\text{accuracy}}(x, y) + r_{\text{format}}(x, y)$$

注意：
- **没有过程奖励（Process Reward Model）**——只看最终结果
- **没有人类偏好数据**——纯规则验证
- 准确性通过数学答案验证或代码测试用例判断

### 2.3 涌现现象

R1-Zero 在 RL 训练过程中自发涌现了以下能力（无任何 CoT 数据！）：

| 涌现能力 | 描述 | 何时出现 |
|---------|------|---------|
| **Chain-of-Thought** | 模型开始生成分步推理 | 训练早期 |
| **反思（Reflection）** | 模型在推理中说"Wait, let me reconsider..." | 训练中期 |
| **自我验证** | 模型检查自己的计算结果 | 训练中期 |
| **回溯** | 发现错误后重新开始推理 | 训练后期 |
| **分解问题** | 将复杂问题拆分为子问题 | 训练后期 |

**关键发现**：这些推理策略**不是从数据中学来的**，而是 RL 的优化压力迫使模型自发发展出来的。

### 2.4 R1-Zero 的「aha moment」

论文中描述了一个令人印象深刻的现象：

```
训练早期: 模型直接输出答案（通常错误）
     ↓
训练中期: 模型开始写推理步骤（准确率上升）
     ↓
训练后期: 模型出现 "Wait, I made an error. Let me redo this."
          这种自我反思完全是自发涌现的！
```

论文称之为模型的 "aha moment"——模型突然「意识到」需要重新检查自己的推理。

### 2.5 R1-Zero 的问题

尽管 R1-Zero 展示了纯 RL 的惊人潜力，但存在明显问题：

| 问题 | 表现 |
|------|------|
| 可读性差 | 推理过程混乱、语言混杂（中英文混合） |
| 格式不一致 | 输出格式不稳定 |
| 过度思考 | 简单问题也生成很长的推理链 |
| 通用能力下降 | 非推理任务（如创意写作）质量下降 |
| 重复循环 | 有时陷入重复推理的死循环 |

这些问题促使团队开发了完整的 R1 训练管线。

## 三、DeepSeek-R1：完整训练管线

### 3.1 四阶段训练流程

R1 的完整训练流程包含四个阶段：

```
Stage 1: Cold Start SFT
  用少量高质量 CoT 数据做 SFT
  → 给模型一个合理的推理起点
  → 解决 R1-Zero 的可读性和格式问题
         ↓
Stage 2: RL Training (GRPO)
  在推理任务上用 GRPO 训练
  → 大幅提升推理能力
  → Reward: 准确性 + 格式 + 语言一致性
         ↓
Stage 3: Rejection Sampling + SFT
  用 Stage 2 的模型生成大量推理数据
  → 用 reward 过滤，保留高质量样本
  → 混合通用 SFT 数据
  → 全量 SFT 训练
  → 恢复通用能力（写作、翻译等）
         ↓
Stage 4: RL Training (Round 2)
  在全部任务上做第二轮 RL
  → 同时优化推理任务和通用对齐
  → 使用 helpfulness + harmlessness reward
```

### 3.2 Stage 1：Cold Start SFT

**目的**：给 GRPO 训练一个好的起点，避免 R1-Zero 的可读性问题。

**数据来源**：
- 数千条高质量长 CoT 样本
- 部分来自人工标注
- 部分来自 R1-Zero 的输出（经人工筛选和编辑）

**关键设计**：
- 数据量很少（几千条 vs SFT 通常需要几万到几十万条）
- 重点是质量而非数量
- 目标是教模型「如何格式化推理过程」，而非教它「如何推理」

### 3.3 Stage 2：GRPO 训练

这是 R1 的核心阶段，使用 GRPO 而非 PPO。

**为什么用 GRPO 而非 PPO？**

| 因素 | PPO | GRPO |
|------|-----|------|
| Critic 网络 | 需要（参数量 = Actor） | 不需要 |
| 671B MoE 的 Critic | 显存不可接受 | 不需要 |
| 实现复杂度 | 高 | 低 |
| 长序列推理 | Critic 估值困难 | 用组内对比，无需估值 |

对于 671B 的 MoE 模型，训练一个同等规模的 Critic 网络是不现实的——GRPO 通过组内采样完全避免了这个问题。

**Reward 设计（Stage 2）**：

| Reward 类型 | 目的 | 实现 |
|------------|------|------|
| 准确性奖励 | 确保推理正确 | 数学：验证最终答案；编程：运行测试用例 |
| 格式奖励 | 确保输出格式正确 | 检查 `<think>...</think>` 标签 |
| 语言一致性奖励 | 防止语言混杂 | 惩罚中英文混合的推理过程 |

注意：**没有使用神经网络 Reward Model**——所有 reward 都是基于规则的。这避免了 Reward Hacking。

### 3.4 Stage 3：Rejection Sampling + SFT

**问题**：Stage 2 的 RL 模型在推理任务上很强，但通用能力（创意写作、翻译等）下降。

**解决方案**：

```
1. 用 Stage 2 的模型生成大量推理数据（数十万条）
2. 用 reward 过滤，保留正确且格式好的样本
3. 将筛选后的推理数据与通用 SFT 数据混合
4. 在 DeepSeek-V3-Base 上重新做全量 SFT
```

**Rejection Sampling 的具体流程**：

```
对于每个推理 prompt x:
  1. 用 RL 模型生成 N 个 response {y_1, ..., y_N}
  2. 用规则 reward 评估每个 response
  3. 只保留 reward 最高的 response(s)
  4. 将 (x, best_y) 加入 SFT 数据集
```

这与 LLaMA 2 的 Rejection Sampling 策略类似（W10 Day1）。

**数据混合比例**：
- 推理数据（数学 + 编程 + 科学）：~60 万条
- 通用 SFT 数据（写作 + 翻译 + 问答等）：~20 万条

### 3.5 Stage 4：第二轮 RL

**目的**：在全量 SFT 模型上做最终的对齐。

**Reward 设计**：
- 推理任务：继续用规则 reward
- 通用任务：使用 Reward Model（helpfulness + harmlessness）

这一阶段更接近传统的 RLHF，但推理任务仍使用规则 reward 而非 RM。

## 四、GRPO 在 R1 中的具体应用

### 4.1 采样策略

对于每个 prompt $x$：
1. 用当前策略 $\pi_\theta$ 生成 $G$ 个 response：$\{y_1, y_2, \ldots, y_G\}$（$G$ 通常为 16~64）
2. 对每个 response 用规则计算 reward：$\{r_1, r_2, \ldots, r_G\}$
3. 在组内计算归一化 advantage
4. 用 GRPO 目标更新策略

### 4.2 长推理序列的处理

R1 的推理序列非常长（可达数千 token），这带来特殊挑战：

| 挑战 | 解决方案 |
|------|---------|
| 序列过长导致 KV Cache 显存爆炸 | 分段生成 + 截断 |
| 长序列中 reward 信号稀疏 | 只用最终结果 reward（不需要过程 reward） |
| Critic 对长序列估值不准 | GRPO 不需要 Critic，完全避免 |
| 生成时间长 | 大规模并行采样 |

### 4.3 训练规模

| 参数 | 值 |
|------|-----|
| 基座模型 | DeepSeek-V3-Base (671B MoE, 37B 激活参数) |
| GRPO 组大小 $G$ | 16~64 |
| 训练 prompt 数量 | 数十万 |
| 最大序列长度 | 32K tokens |
| 训练硬件 | 数千张 GPU |

## 五、推理涌现的深入分析

### 5.1 为什么 RL 能诱导推理能力

| 视角 | 解释 |
|------|------|
| **优化压力** | 为了获得 accuracy reward，模型被迫发展出能提高正确率的策略（即推理） |
| **探索机制** | GRPO 的多组采样让模型探索不同的回答方式，推理链被发现是高 reward 的策略 |
| **预训练知识** | 预训练阶段模型已经「见过」推理文本，RL 只是激活和强化了这种能力 |
| **涌现性质** | 推理能力可能是大模型的涌现属性，RL 提供了合适的优化信号来解锁它 |

### 5.2 R1-Zero vs R1 的推理质量对比

| 维度 | R1-Zero | R1 |
|------|---------|-----|
| 推理正确率 | 较高（但不稳定） | 高且稳定 |
| 推理可读性 | 差（语言混杂、格式混乱） | 好（结构化、清晰） |
| 通用能力 | 下降 | 保持（Stage 3 恢复） |
| 过度思考 | 严重（简单题也长推理） | 改善 |
| 重复循环 | 偶尔出现 | 基本消除 |

### 5.3 思考深度的 Scaling

R1 发现了一个有趣的现象：**推理链的长度在 RL 训练过程中自然增长**。

```
训练早期: 推理链 ~200 tokens  → 准确率 ~40%
训练中期: 推理链 ~500 tokens  → 准确率 ~60%
训练后期: 推理链 ~2000 tokens → 准确率 ~80%
```

模型在 RL 训练中自动学会了「难题需要更多思考步骤」——这是一种 test-time compute scaling 的自然涌现。

## 六、数据蒸馏策略

### 6.1 蒸馏的动机

R1 (671B MoE) 太大，无法在普通硬件上部署。解决方案：用 R1 生成的推理数据训练小模型。

### 6.2 蒸馏流程

```
Step 1: 准备推理 prompt 集合
Step 2: 用 R1 生成高质量推理过程
Step 3: 用 reward 过滤，保留正确答案的推理链
Step 4: 将 (prompt, 推理过程, 答案) 作为 SFT 数据
Step 5: 在小模型上做 SFT 训练
```

### 6.3 蒸馏结果

| 蒸馏目标模型 | 基座 | AIME 2024 | MATH-500 |
|------------|------|-----------|----------|
| DeepSeek-R1-Distill-Qwen-1.5B | Qwen2.5-1.5B | 28.9% | 83.9% |
| DeepSeek-R1-Distill-Qwen-7B | Qwen2.5-7B | 55.5% | 92.8% |
| DeepSeek-R1-Distill-Qwen-14B | Qwen2.5-14B | 69.7% | 93.9% |
| DeepSeek-R1-Distill-Qwen-32B | Qwen2.5-32B | 72.6% | 94.3% |
| DeepSeek-R1-Distill-LLaMA-8B | LLaMA-3.1-8B | 50.4% | 89.1% |
| DeepSeek-R1-Distill-LLaMA-70B | LLaMA-3.3-70B | 70.0% | 94.5% |

蒸馏模型的表现远超同规模的基座模型，证明推理能力可以通过数据蒸馏有效传递。

### 6.4 蒸馏 vs 直接 RL

论文做了一个重要对比：对小模型做直接 RL 训练 vs 用 R1 数据蒸馏。

| 方法 | 7B 模型 AIME | 说明 |
|------|------------|------|
| 直接 RL (GRPO) | ~30% | 小模型的 RL 探索能力有限 |
| R1 数据蒸馏 (SFT) | ~55% | 直接学习 R1 的推理模式 |

结论：**对于小模型，蒸馏比直接 RL 更有效**。因为小模型在 RL 训练中的探索空间有限，难以自发涌现复杂推理能力。

## 七、R1 的技术要点总结

### 7.1 关键技术决策

| 决策 | 选择 | 原因 |
|------|------|------|
| RL 算法 | GRPO（非 PPO） | 避免 671B 模型的 Critic 开销 |
| Reward | 规则验证（非 RM） | 避免 Reward Hacking，数学答案可精确验证 |
| 训练起点 | 预训练 Base 模型 | 证明推理能力可以从 RL 中涌现 |
| Cold Start | 少量 CoT SFT 数据 | 解决 R1-Zero 的格式和可读性问题 |
| 通用能力恢复 | Rejection Sampling + SFT | 解决 RL 后通用能力下降的问题 |
| 小模型部署 | 数据蒸馏 | 比直接 RL 更有效 |

### 7.2 R1 对领域的影响

| 影响 | 说明 |
|------|------|
| 证明 RL 可以诱导推理 | 推理能力不一定需要 CoT 标注数据 |
| GRPO 的工程价值 | 在超大模型上替代 PPO 的实用方案 |
| 规则 reward 的价值 | 在可验证任务上不一定需要 RM |
| 蒸馏的有效性 | 大模型的推理能力可以通过蒸馏传递 |
| 开源推动 | 首个公开训练细节的 o1 级别模型 |
