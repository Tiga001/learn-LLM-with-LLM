> **目标**：系统理解 RAG 的完整链路：离线索引、在线检索、上下文增强与答案生成；掌握文档切分、向量化、召回、重排和生成之间的关系；理解 Naive RAG 的常见失败模式与主流优化方向；为 Day6 手写 RAG 系统实践做准备。

---

## 一、为什么需要 RAG

### 1.1 仅靠参数记忆不够

LLM 只靠参数存储知识会遇到三个问题：

1. 知识会过期
2. 私有数据不在预训练语料中
3. 很难精确追溯信息来源

例如：

- 公司内部文档
- 最新产品说明
- 教学资料更新
- 法规条款变动

这些内容通常更适合“外部存储 + 按需检索”，而不是要求模型全部背下来。

### 1.2 RAG 的核心思想

RAG 的思想可以概括为一句话：

> 模型不需要记住一切，但要知道在回答前去哪里找。

标准流程如下：

```text
文档库
  ↓
切分 chunk
  ↓
向量化 embedding
  ↓
建立索引

用户提问
  ↓
问题向量化
  ↓
相似度检索
  ↓
取回相关 chunk
  ↓
把 chunk 拼到 prompt
  ↓
LLM 生成答案
```

---

## 二、RAG 的两条主链路

### 2.1 离线链路：索引构建

离线阶段通常做一次或周期性更新，步骤包括：

1. 读取原始文档
2. 清洗文本
3. 切分为多个 chunk
4. 用 embedding 模型编码
5. 写入向量索引或数据库

这一阶段的目标是把“难检索的长文档”转换成“可高效召回的知识单元”。

### 2.2 在线链路：检索增强生成

在线阶段每次用户提问都会运行：

1. 对 query 向量化
2. 在索引中召回 top-k chunk
3. 可选重排
4. 构造增强 prompt
5. 调用生成模型输出答案

这两条链路一个偏“数据准备”，一个偏“实时问答”。

## 三、文档切分为什么重要

### 3.1 太长或太短都不好

切分过长会导致：

- 召回粒度太粗
- chunk 里噪声多
- prompt 被无关内容污染

切分过短会导致：

- 语义上下文被打断
- 检索结果缺少必要背景
- 需要更多 chunk 才能回答问题

### 3.2 常见切分策略

#### 固定长度切分

```text
每 256 / 512 token 切一段
```

优点：

- 实现简单
- 工程稳定

缺点：

- 容易切断语义边界

#### 滑窗切分

```text
chunk_size = 500
overlap = 100
```

优点：

- 缓解边界切断问题
- 提高召回连续语义的概率

缺点：

- 索引体积变大
- 重复内容更多

#### 结构化切分

按标题、段落、列表、代码块等结构切分。

优点：

- 更符合文档本身语义
- 特别适合教程、文档、法律条文

缺点：

- 实现更复杂
- 依赖格式质量

## 四、向量化与相似度检索

### 4.1 embedding 的作用

embedding 模型把文本映射到一个向量空间：

$$
q = f_{\text{embed}}(\text{query}), \quad d_i = f_{\text{embed}}(\text{chunk}_i)
$$

如果 query 和某个 chunk 语义相关，那么它们的向量应该更接近。

### 4.2 常见相似度函数

#### 点积

$$
\text{score}(q, d) = q^T d
$$

#### 余弦相似度

$$
\text{score}(q, d) = \frac{q^T d}{\|q\| \|d\|}
$$

#### L2 距离

$$
\text{dist}(q, d) = \|q - d\|_2
$$

实际工程中最常见的是余弦相似度或归一化后的点积。

### 4.3 ANN：近似最近邻搜索

文档规模大时，不能每次都和全部向量暴力比较，因此需要近似最近邻搜索：

- FAISS
- HNSW
- IVF
- PQ

RAG 的目标不是找绝对最优邻居，而是在延迟可控的前提下找到足够好的候选集合。

## 五、从召回到生成

### 5.1 Naive RAG 的 prompt 结构

一个最简单的 RAG prompt 如下：

```text
你是一个问答助手，请基于给定资料回答问题。

资料 1:
...

资料 2:
...

用户问题:
...

要求:
1. 优先基于资料作答
2. 若资料不足，明确说明
```

### 5.2 为什么只“检索”还不够

即使 top-k 召回看似不错，生成阶段仍可能出问题：

- chunk 太多，关键信息被淹没
- chunk 顺序不合理
- query 改写不充分
- 模型忽视检索结果，继续幻觉

因此一个高质量 RAG 系统不仅要“召回来”，还要“喂得对”。

## 六、Naive RAG 的常见失败模式

### 6.1 Recall 不足

问题：

- top-k 里根本没有正确答案

原因可能是：

- embedding 模型不合适
- 切分太差
- query 表达和文档表述不一致

### 6.2 Precision 不足

问题：

- 召回了很多无关 chunk

后果：

- prompt 噪声大
- 模型更容易答偏

### 6.3 Context Dilution

即使 top-k 中有正确 chunk，如果同时塞进太多相近但不相关的文本，模型也可能抓不到重点。

### 6.4 Hallucination

模型在资料不足时仍强行编答案，是最典型的 RAG 失败。

因此系统提示词中应明确：

- 资料不足时要承认不知道
- 不允许脱离证据瞎编

## 七、RAG 的主流优化方向

### 7.1 Query 改写

把用户原问题改写成更适合检索的形式。

例如：

```text
原问题: 它为什么快？
改写后: vLLM 为什么推理吞吐更高？
```

### 7.2 Hybrid Retrieval

把稠密检索和关键词检索结合：

- Dense Retrieval：更强语义能力
- BM25 / Keyword：更强精确匹配

两者结合能提升鲁棒性。

### 7.3 Reranking

先用快方法召回 top-20，再用更强的 reranker 重排 top-5。

这样既兼顾速度，也提高精度。

### 7.4 压缩与上下文选择

不必把所有召回 chunk 原样塞进 prompt，可以做：

- 摘要压缩
- 去重
- 关键句抽取
- 按问题相关性排序

### 7.5 多跳检索

复杂问题往往需要多步检索：

```text
问题 -> 第一次检索 -> 得到中间实体 -> 第二次检索 -> 最终回答
```

这类场景已经和 Agent 开始融合。

## 八、RAG 与微调、Agent 的关系

### 8.1 RAG vs 微调

RAG 更适合：

- 知识频繁变化
- 私有数据接入
- 需要可追溯引用

微调更适合：

- 改变输出风格
- 学习任务格式
- 习得长期稳定行为模式

两者并不冲突。

### 8.2 RAG 与 Agent 的连接

当检索不再只是“查一次资料”，而是需要：

- 改写 query
- 多轮检索
- 调用多个数据源
- 结合工具结果决策

RAG 就会自然演化为 Agent 的一个子模块。
