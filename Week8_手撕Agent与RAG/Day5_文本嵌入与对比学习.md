# Day 5：文本嵌入与对比学习 — RAG 检索效果的数学基础

> **目标**：理解文本嵌入模型为什么能把语义相近的文本映射到相近向量；掌握双塔架构、相似度函数与 InfoNCE 对比学习损失；理解正负样本构造、in-batch negatives 和硬负样本的重要性；建立从“embedding 训练”到“RAG 检索效果”的完整联系。

---

## 一、为什么需要文本嵌入

### 1.1 稀疏匹配的局限

传统关键词检索依赖字面重合：

```text
Query: "大模型如何外接知识库"
文档: "检索增强生成能够把外部文档接入语言模型"
```

虽然语义接近，但字面词可能不一致，导致关键词匹配效果有限。

### 1.2 embedding 的目标

embedding 模型希望学习一个映射函数：

$$
f(x) \in \mathbb{R}^d
$$

让语义相近的句子在向量空间里靠近，语义不相关的句子彼此远离。

理想情况下：

$$
\text{sim}(f(q), f(d^+)) > \text{sim}(f(q), f(d^-))
$$

其中：

- $q$ 是 query
- $d^+$ 是相关文档
- $d^-$ 是无关文档

## 二、文本嵌入是怎么得到的

### 2.1 从 Transformer 到句向量

一个编码模型经过多层 Transformer 后，会输出每个 token 的隐藏表示：

$$
H = [h_1, h_2, \dots, h_n], \quad h_i \in \mathbb{R}^d
$$

要得到整句向量，通常需要做 pooling。

### 2.2 常见 pooling 方式

#### CLS pooling

取 `[CLS]` 位置的向量：

$$
z = h_{\text{CLS}}
$$

#### Mean pooling

对全部 token 向量取平均：

$$
z = \frac{1}{n} \sum_{i=1}^n h_i
$$

#### Max pooling

对每一维取最大值。

在 Sentence-BERT 一类模型中，mean pooling 通常比直接取 CLS 更稳定。

---

## 三、双塔架构（Bi-Encoder）

### 3.1 为什么检索常用双塔

双塔结构把 query 和文档分别编码：

$$
q = f_q(\text{query}), \quad d = f_d(\text{doc})
$$

然后在向量空间里做相似度比较。

```text
query --> encoder_q --> q_vec
doc   --> encoder_d --> d_vec
score = sim(q_vec, d_vec)
```

优点：

- 文档向量可离线预计算
- 检索速度快
- 适合大规模 ANN 搜索

### 3.2 双塔与交叉编码器的区别

#### 双塔（Bi-Encoder）

```text
query 和 doc 分别编码
```

优点：

- 快
- 可扩展

缺点：

- query 和 doc 之间交互较弱

#### 交叉编码器（Cross-Encoder）

```text
[query ; doc] 拼接后一起编码
```

优点：

- 交互更强
- 排序精度高

缺点：

- 无法离线预计算
- 推理慢

因此工程上常见组合是：

```text
双塔召回 + 交叉编码器重排
```

## 四、相似度函数

### 4.1 点积

$$
\text{sim}(q, d) = q^T d
$$

简单高效，但受向量范数影响较大。

### 4.2 余弦相似度

$$
\text{sim}(q, d) = \frac{q^T d}{\|q\| \|d\|}
$$

去除了长度影响，更关注方向一致性。

### 4.3 温度缩放

在对比学习里，经常会加入温度参数 $\tau$：

$$
\text{score}(q, d) = \frac{q^T d}{\tau}
$$

温度越小，softmax 分布越尖锐，模型会更强调“最相似”的样本。

## 五、InfoNCE：对比学习的核心损失

### 5.1 基本形式

给定一个 query $q$、一个正样本 $d^+$、若干负样本 $d_1^-, d_2^-, \dots, d_k^-$，InfoNCE 损失为：

$$
\mathcal{L} = - \log \frac{\exp(\text{sim}(q, d^+) / \tau)}{\exp(\text{sim}(q, d^+) / \tau) + \sum_{j=1}^k \exp(\text{sim}(q, d_j^-) / \tau)}
$$

它的目标很直观：

- 提高正样本相似度
- 降低负样本相似度

### 5.2 从分类角度理解

可以把“在候选文档中找正样本”看成一个多分类问题：

- 正样本是正确类别
- 其他候选都是错误类别

因此 InfoNCE 本质上是在做 softmax 分类，只不过分类对象是“相似度分数”。

### 5.3 为什么它适合检索

检索任务的核心目标正是：

> 在一堆候选文档中，把真正相关的文档排到最前面。

InfoNCE 直接优化了这个目标的近似形式。

## 六、正负样本怎么构造

### 6.1 正样本

正样本一般来自：

- 问答对中的标准文档
- query 对应的点击文档
- 标注好的相关段落

### 6.2 随机负样本

从别的 query 的文档里随机采样。

优点：

- 简单

缺点：

- 太容易区分
- 训练信号可能偏弱

### 6.3 硬负样本（Hard Negatives）

硬负样本指看起来很像正确答案、但其实不是的样本。

例如：

```text
Query: “RAG 的核心流程是什么？”

Positive:
  “RAG 包括切分、向量化、索引、检索、生成”

Hard Negative:
  “Agent 包括规划、调用工具、观察结果”
```

硬负样本能显著提升模型区分近义概念的能力。

### 6.4 In-batch Negatives

在一个 batch 中，其他样本的正例可以天然充当当前 query 的负例。

假设 batch 大小为 $B$，那么每个 query 可得到：

$$
B - 1
$$

个免费负样本。

这就是大 batch 在对比学习里很重要的原因。

## 七、训练 embedding 模型时的关键设计

### 7.1 向量归一化

很多检索系统会对向量做 L2 归一化：

$$
\hat{z} = \frac{z}{\|z\|}
$$

这样点积就等价于余弦相似度，训练和检索更加一致。

### 7.2 batch 大小

batch 越大，in-batch negatives 越多，通常检索效果更好。

### 7.3 温度参数

温度太大：

- 相似度分布太平
- 正负样本区分不明显

温度太小：

- 梯度过于尖锐
- 训练可能不稳定

### 7.4 数据质量

embedding 模型的上限很大程度由数据决定：

- 正样本是否真的相关
- 负样本是否足够难
- query 是否覆盖真实用户表达

---

## 八、从 embedding 到 RAG 效果

### 8.1 检索质量直接决定生成上限

如果 top-k 根本没召回正确文档，那么生成模型再强也无能为力。

可以近似理解为：

$$
\text{RAG 质量} \approx \text{召回质量} \times \text{生成质量}
$$

虽然这不是严格公式，但它强调了一件事：

> 检索层是 RAG 的上游瓶颈。

### 8.2 为什么“看起来能检索”不等于“真的好用”

很多 embedding 模型在简单例子上表现不错，但在真实系统里会遇到：

- 长尾 query
- 术语缩写
- 领域概念冲突
- 相近概念区分困难

因此必须用真实评测集来判断 embedding 质量。

## 九、双塔、重排与 RAG 系统的分工

在完整系统中，常见分工如下：

```text
双塔 embedding:
  负责快速粗召回

reranker:
  负责精排

LLM:
  负责根据检索结果组织答案
```

如果把三者混为一谈，系统很难定位瓶颈。
